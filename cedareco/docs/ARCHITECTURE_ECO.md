# 瓶中生态（Eco）架构

本文描述当前 `eco/` 游戏本体及 `eco_adapter/` 的实际边界。代码是唯一事实来源；机制提案与既有说明只用于理解意图，若与代码不一致，以 `engine.py`、`standalone_server.py` 和 `eco_adapter/handler.py` 为准。

## 1. 总览

Eco 是一个按天推进、可确定性重放的池塘模拟。核心规则集中在 `engine.py`：物种参数、环境演化、天气、灾害、访客与定居者、指令解析、只读 JSON 投影和人类协作入口均在同一模块中。平台适配器不实现生态规则，只负责 MCP 参数转换、玩家隔离、事务与持久化。

时间单位固定为：1 回合 = 1 天，1 季 = 30 天，1 年 = 120 天。随机源是可序列化的 `Mulberry32`；`rng_state` 随存档保存，因此相同 seed 和相同指令序列通常可复现同一条随机流。例外是 `_return_boost_active()` 读取 Asia/Shanghai 日期：不晚于 2026-07-10 时定居者回归概率被强制为 1；该门现已过期，但复现截止日前的运行必须考虑墙钟差异。

### 文本版模块依赖图

```text
平台 MCP / 平台 HTTP API
          |
          v
eco_adapter/handler.py -----------------> data/sessions.db
  参数校验、玩家存档、事务、锁、通知          eco_sessions.save_data
          |
          v
eco/engine.py <-------------------------- 游戏规则唯一实现
  |       |  \
  |       |   +--> api_state / api_codex / api_folio / api_annals / api_species
  |       +------> human_action（六种人类协作）
  +--------------> cmd / new_game / tick
          ^
          |
eco/standalone_server.py <---- eco/web/*、standalone_client.py
  单池 JSON 存档、HTTP、静态资源

eco/build_blind.py --读取--> engine.py --生成--> ecosystem.py
eco/scripts/verify_*.py -----------------------> engine / standalone_server
```

依赖方向有两个约束：`engine.py` 不依赖适配器或网页；`ecosystem.py` 是生成物，不是第二份规则实现。

## 2. 模块结构

| 路径 | 职责 | 维护约束 |
|---|---|---|
| `engine.py` | 全部游戏规则、状态迁移、命令、投影和协作接口 | 规则事实源；状态字段变化必须同步 `fresh_state()` 与 `_migrate()` |
| `ecosystem.py` | 将引擎源码以 base64 封装的盲玩入口，只导出 `cmd()`、`new_game()` | 由 `build_blind.py` 生成，禁止手改 |
| `build_blind.py` | 生成或用 `--check` 校验 `ecosystem.py` | 修改引擎后必须重新生成并校验 |
| `standalone_server.py` | 单池独立 HTTP 服务、原子 JSON 存档、静态前端 | 通过 `PondStore` 串行调用引擎；不复制规则 |
| `standalone_client.py` | 独立服务的命令行 HTTP 客户端 | 只负责请求封装 |
| `web/`、`assets/` | 状态/图鉴/年鉴界面与六个协作小游戏 | 小游戏只产出 action payload，最终判定在引擎 |
| `scripts/` | 人类协作、天气文案、定居者亲缘/昵称、独立服务的回归校验 | 是机制边界的可执行断言 |
| `eco_adapter/handler.py` | 平台 MCP 工具、per-player SQLite 存档、并发控制、平台通知 | 位于游戏仓库外；不得承载生态数值规则 |

`README.md`、`API.md`、`MECHANICS_V2.md` 等是使用或设计说明，不参与运行时。

## 3. 运行入口与状态所有权

### 3.1 引擎入口

- `new_game(seed)`：创建 `fresh_state(seed)`，赋给进程级 `_STATE`，并调用默认文件存档。
- `cmd(command)`：加载 `_STATE`，按中英文分号拆成多条指令，经 `_exec_one()` 分发；结束时调用 `save_state()`。
- `tick(state)`：推进一天，是所有自然演化的唯一入口。`observe` 调一次，`wait` 最多连续调用七次，并在关键事件或 `pending_choice` 时暂停。
- `human_action(state, action, payload)`：在当前天修改传入状态，不推进回合，不自行写盘。
- `api_*()`：把内部状态投影成前端可读 JSON，不推进游戏。

默认 CLI 模式由引擎读写 `eco_save.json`。平台模式和独立服务都接管持久化，不应依赖该文件。

### 3.2 存档模型

`fresh_state()` 返回版本为 2 的单个 JSON 对象，主要分区如下：

- 时间与随机：`seed`、`rng_state`、`turn`、`season`、`weather`、梅雨计数。
- 生态数值：`populations`、六项 `env`、变态年龄池 `cohorts`。
- 长期个体：`settlers`、稳定 `settler_id`、亲缘、昵称、健康、年龄、离塘与回归档案。
- 事件状态：`flags`、`chain`、`active_weather`、`diseases`、`pending_choice`。
- 叙事与解锁：`unlocked_species`、`seen`、`folio`、`chronicle`、`achievements`。
- 展示缓存：`prev_env`、`last_delta`、最近 30 天 `history`、当日 `log`。
- 协作交接：`pending_human_notices`，供下一次 `cmd()` 读取并清空。

`_migrate()` 按字段补齐旧档，并处理福寿螺、巴西龟、灾害、定居者 ID/亲缘等历史结构。新字段不能只加在业务分支中，否则旧档会缺失。

## 4. 物种与数值体系

### 4.1 两类生命模型

种群与定居者是两套模型：

1. `SPECIES` 管理连续种群。当前共 24 种，其中 21 种没有 `hidden` 标记（仍受解锁控制），隐藏种为细菌、蜻蜓成虫、蚊子。按营养级分为 4 个生产者、12 个初级消费者、6 个次级消费者、1 个顶级消费者和 1 个分解者。
2. `SETTLER_TYPES` 管理个体。水蛇、流浪乌龟、螃蟹、野鸭、翠鸟、苍鹭各自拥有年龄、健康、食谱、幼体期和同种上限，不计入 `populations`。

可见种群为：水藻、浮萍、芦苇、水蚤、田螺、蝌蚪、孑孓、蜻蜓幼虫、鲫鱼、青蛙、鲤鱼、草鱼、鳑鲏、泥鳅、淡水虾、河蚌、水黾、睡莲、蟾蜍、水蚯蚓、田鼠。新局是空池，只有细菌初始为 100；水藻、浮萍、芦苇只是最初可召唤名册，不会自动入池。

每个 `SPECIES` 条目至少定义空间、营养级、出生率、死亡率、承载量、食物和捕食效率；可选字段实现需氧、碎屑摄食、滤食、排泄、翻泥、生命周期等特性。内部数量保留浮点，对外由 `_ipop()` 取整显示。

### 4.2 环境闭环

`env` 是种群之间的共同介质：

| 指标 | 新局值 | 主要来源与去向 |
|---|---:|---|
| `water_temp` | 18.0℃ | 向季节目标缓动，再叠加每日天气与持续天气 |
| `dissolved_oxygen` | 9.0 mg/L | 温度饱和值、气体交换、光合作用和风雨增加；结冰、覆盖、分解与绿潮消耗 |
| `light` | 0.85 | 季节目标 × 天气 × 雾/结冰 × 水面覆盖 × 浑浊度 |
| `nutrients` | 15.0 | 降雨、排泄、分解增加；植物生长和河蚌滤食消耗 |
| `detritus` | 0.0 | 死亡、落叶、投喂、定居者增加；碎屑摄食与细菌分解消耗 |
| `turbidity` | 0.05 | 每日自然沉降；降雨、翻泥、灾害和拔草增加；河蚌降低 |

水面覆盖率由浮萍/280 + 睡莲/80，再加显形后的水葫芦覆盖组成，最后夹在 0..1。覆盖达到 0.7 时每天额外耗氧 0.3；达到 0.9 时耗氧 1.0 且光照上限为 0.1。

### 4.3 每日数值结算

`tick()` 顺序本身是规则，不能随意调换：

1. 保存上一日快照，`turn += 1`，计算季节并抽取每日天气；处理季节切换。
2. 更新温度、光照、浑浊、营养盐、溶氧、结冰和冬眠。
3. 推进 `active_weather` 与外来入侵。
4. 生产者按 Logistic 模型增长；承载量由光、营养盐、温度、季节和干旱修正。
5. 汇总捕食意图后统一结算；每个捕食者—猎物结算项最多取当时猎物量的 50%，捕食量的 15% 转为捕食者增长。
6. 消费者按季节、温度、食物份额和承载量增长，再结算自然死亡、缺氧与物种特例。
7. 应用滤食/翻泥/排泄等特殊效果，再推进疫病和生物灾害。
8. 推进变态年龄池和季节产卵。
9. 细菌把碎屑转为营养盐并消耗氧；随后检查翻塘与富营养化萌藻。
10. 结算定居者摄食、健康、离塘、繁殖与互动。
11. 更新解锁、万物志与自然恢复，再抽访客、决策和环境事件。
12. 检查暂停、成就和年报，记录变化量、30 天历史和新 `rng_state`。

生产者的核心形式为 `growth = birth_rate × 各乘数 × n × (1 - n/K)`；消费者还以共享食物份额把繁殖因子压在 0.2..1.0。捕食量近似为 `predation × 温度因子 × 捕食者 × 猎物`，再受食物特效和上限约束。死亡会回流为碎屑，细菌再将碎屑转成营养盐，从而构成“死亡 → 分解耗氧 → 肥水 → 植物增长”的闭环。

需氧物种在溶氧低于 4 时增加死亡；鱼类低于 3 时死亡率再乘 3。溶氧低于 2 且碎屑高于 150 会杀死 40% 需氧生物；低于 1.5 的首次跨越还会让五类鱼再损失 50%，直到溶氧恢复到 3 才解除去重标记。

### 4.4 生命周期、解锁与评分

- 蝌蚪 30 天后按 0.4 比例变为青蛙；孑孓 10 天后按 0.6 变为蚊子；蜻蜓幼虫 60 天后仅在夏季按 0.5 羽化。年龄按 `cohorts` 批次追踪。
- 青蛙在春季产蝌蚪，蜻蜓成虫在秋季产幼虫。青蛙冬季移入 `hibernate`，春融时恢复。
- 造物名册以历史峰值和生态条件解锁；两次解锁至少间隔 5 天，每回合最多一种。已解锁且归零的非季节性物种在食物条件满足时每天有 1% 概率自然恢复 2..5 个体。
- 定居者在种群结算后摄食。健康决定饥饿离塘，年龄上限可由高健康天数延长；春夏有兼容成体对时每天 5% 繁殖，每种每年最多一只幼体，并检查亲子/同胞关系。
- `_pond_score()` 按多样性 20%、Shannon 均衡度 15%、四营养级完整度 20%、环境健康 15%、健康定居者 10%、持续天数 20% 加权；溶氧低于 3/1 时总分再乘 0.5/0.3。

## 5. 天气状态机

代码中有三种会影响天气的状态，彼此不可混写：

1. `weather`：每天一抽的短天气，作用于当天环境、出生率、死亡率和文案。
2. `active_weather`：跨日持续过程，当前支持干旱、热浪、暴雨、洪水、下雨；与当日 `weather` 叠加，不互斥。
3. `chain`：带截止回合的后效，如 `drought`、`heatwave`、`fog`、`ice_hole`、`waterbloom_x2`。`_chain_set(key, days)` 存 `turn + days`，`_chain_active()` 使用严格的 `turn < deadline`；若在当日对应结算阶段之后才写入，后续完整生效回合通常是 `days - 1`。

### 5.1 每日天气

`_roll_weather()` 的优先级是：进行中的梅雨 > 季节概率表 > 春寒/秋霜替换。

```text
每个 tick 开头
  |
  +-- meiyu_remaining > 0
  |      -> remaining - 1
  |      -> meiyu_days + 1
  |      -> 子天气在“阴/小雨”间交替，weather = 梅雨
  |
  +-- 否则按当前季节概率抽取
         |
         +-- 抽到梅雨 -> 随机持续 3..7 天，当天即为第 1 天
         +-- 其他天气 -> 春季前 10 天有 5% 替换为春寒；
                         秋季后 5 天有 10% 替换为秋霜
```

梅雨只能在春季第 25..30 天或夏季第 1..10 天起头；窗口外，梅雨的 5% 概率并入阴天。各季概率总和均为 100：

| 季节 | 概率表（%） |
|---|---|
| 春 | 晴25、多云22、阴18、小雨13、雨7、雾5、大风5、梅雨5 |
| 夏 | 晴40、多云18、阴12、小雨10、雨8、雾2、大风5、梅雨5 |
| 秋 | 晴30、多云22、阴18、小雨10、雨5、雾8、大风7 |
| 冬 | 晴25、多云23、阴18、小雨5、雾4、大风8、雪12、大雪5 |

`WEATHER_ENV_MOD` 为每种天气定义“光照乘数、温度增量、溶氧增量、营养盐增量、浑浊度增量”；`WEATHER_BIRTH`、`WEATHER_DEATH` 只覆盖有特例的物种。梅雨超过 5 天后还会抑制水藻、促进孑孓并把疫病起始概率乘 1.2。

### 5.2 季节与结冰

季末最后 3 天的目标水温线性过渡到下一季。冬季水温低于 4℃时 `ice_on=True`，持续至春季：结冰每天额外耗氧 0.5，无有效冰洞的天数记入 `ice_suffocation`。春融后，冬眠青蛙按 `ice_suffocation / ice_total_days × 40%` 折损，再清空本冬计数。

### 5.3 持续天气

持续天气由 `_start_weather()` 写成 `{kind, elapsed, duration, mitigated}`，时长由当前回合确定性派生，不额外消耗随机数。`_process_weather()` 每天先加一日，再应用效果；到期输出结束事件并置空。

| 过程 | 时长 | 主要逐日效果 |
|---|---:|---|
| 干旱 | 5..7 天 | 升温、降氧、加营养盐；`drought` chain 有效时所有承载量减半 |
| 热浪 | 3..5 天 | 逐日升温、每天降氧 1.5；末日低氧可触发翻塘 |
| 暴雨 | 1..2 天 | 增浑浊 0.2、营养盐 22；提前加固时效果减半 |
| 洪水 | 2..3 天 | 增浑浊 0.3、营养盐 30、碎屑 25 |
| 下雨 | 3..5 天 | 降温、逐步复氧、营养盐小幅增加；是干旱选择的恢复路线 |

`mitigated=True` 会把热浪或暴雨的逐日压力减半；干旱选择“下雨”走的是独立的 `下雨` 过程。

## 6. 灾害触发链

### 6.1 总体调度

灾害分为四组：

- 随机环境/入侵：在 `_random_events()` 中触发。每季最多三次计数灾害；持续天气与巴西龟/福寿螺入侵之间不叠开；未决 `pending_choice` 会阻止新的决策。
- 持续天气：玩家 `choose` 后才由 `_resolve_choice()` 启动，实际逐日伤害从后续 `tick()` 的 `_process_weather()` 开始。
- 生物灾害：绿潮、鼠患、孑孓爆发由 `_process_biological_disasters()` 按数值阈值自动触发，不使用每季三次上限。
- 疫病/翻塘：疫病由高密度概率触发；翻塘由低氧和碎屑链直接触发，均独立于 `pending_choice`。

随机事件还会写入 `chain` 形成后继：暴雨在碎屑高时写入 `waterbloom_x2`，洪水写入 `crab_incoming`，热浪写入 `heatwave`，大雾写入 `fog`。这些后效的实际回合数服从上一节的严格截止语义：例如随机事件阶段设置两天 `fog`，会影响下一次环境结算；`crab_incoming` 在后续有效回合检查螃蟹决策。暴雨还会写入一天的 `crab_maybe`，但它在下一次随机事件检查前已到严格截止点，当前代码不会进入该分支。注意“水华”是随机决策事件，“绿潮”是水藻/营养盐阈值触发的协作灾害，二者不是同一状态。

### 6.2 六条人类协作灾害链

| 灾害 | 触发与状态 | 自然/小机链路 | 人类动作 |
|---|---|---|---|
| 巴西龟 | 春夏、满足灾害调度条件时每天 0.5%；`active -> expelled_once -> active/gone` | 活跃时浮萍×0.90、田螺×0.85、鳑鲏×0.90；第一次驱赶后 1..3 天回访，人机累计两次才 `gone` | `expel_turtle` 与小机共用累计器；仅 active 窗口可用 |
| 福寿螺 | 春夏每天 0.5% 先进入 `incubating`，固定第 3 天孵化为 12 只 active | 剩余只数把水藻出生率线性压至最低 0.7；小机可手捞清零，或投螃蟹进入 clearing、每天吃 2 只 | `catch_snail` 可在孵化前清卵（碎屑+3），或爆发后按只捞；同场一次，清零碎屑+10 |
| 水葫芦 | 春季每天 0.3%；潜伏 3 天后出现决策 | 此后覆盖每天+0.04、上限0.75；草鱼每天最多削减 `0.004 × min(数量,8)`；小机拔除代价按剩余/爆发覆盖缩放 | `pull_hyacinth` 同场一次，最多7株、每株覆盖-0.02，浑浊+0.05 |
| 鼠患 | 田鼠>15 且冷却已过，持续3天 | 每天芦苇乘 `1 - 田鼠×0.012`，浑浊+田鼠×0.008；降至爆发数25%立即平息，否则到期田鼠×0.75并冷却10天 | `hunt_rat` 同场一次，最多打12只；跨过25%线当场解除 |
| 绿潮 | 水藻>500 且营养盐>150，持续随机3..5天 | 每天耗氧0.8；结束时移除状态 | `skim_algae` 同场一次，最多捞50；按累计捞量最多减免每天0.3耗氧，达到50再缩短1天 |
| 结冰 | 冬季水温<4℃，直到春融 | 每天耗氧并累计无洞憋气；小机 `crack` 另耗5碎屑 | `crack_ice` 每人每天一次；当天第一方生成共享随机值，单方成功率40%，双方同日复用该值并提升为85%；成功冰洞作用后续3个 tick，每天复氧0.3且不记憋气 |

人类动作除凿冰首次判定外不消耗主随机流，全部不推进天数。成功后总会写年鉴，并把给小机的文字放入 `pending_human_notices`；巴西龟、爆发后的福寿螺、水葫芦、鼠患和绿潮分支还会补充灾害志，孵化前清卵与凿冰不会。下一次 `cmd()` 在执行指令前只消费一次通知。若人类已处理掉与当前 `pending_choice` 同名的灾害，`_human_clear_pending()` 会取消该选择并返回被打断的 wait 天数。

### 6.3 其他灾变

- 疫病：某可见种群超过基础承载量 80%、不在免疫期时，每天 3% 起病；持续 7..10 天。第 1..2 天仅前兆，第 3..5 天死亡倍率 2 且有 30% 沿捕食/同食物邻接传播，第 6 天起倍率 3；结束后该物种免疫 30 天。`sanitized` 会把传播概率减半。
- 孑孓爆发：孑孓>200 且蜻蜓幼虫<10；条件解除即结束。
- 水华决策：热浪、夏季高营养盐或高藻量按各自概率触发，玩家可紧急清理或等待；它不写入 `bio_disasters["绿潮"]`。
- 鸬鹚、寒潮、孩童投石等为即时灾害；洪水、暴雨、干旱、热浪使用持续过程。

## 7. 协作玩法与前端

`web/app.js` 根据 `api_state()` 暴露的灾害状态和服务端补充的 `available_human_actions` 决定是否显示小游戏。六个小游戏只负责交互并提交：

- `hunt_rat {count}`
- `expel_turtle {}`
- `catch_snail {count}`
- `pull_hyacinth {stalks}`
- `skim_algae {amount}`
- `crack_ice {}`

引擎再次校验灾害是否活跃、是否已帮助、参数类型和上限；不能把前端结果当作可信状态。成功响应包含 `ok`、`action`、`message`、`events`、`summary`，失败包含稳定错误码和说明。

独立模式下，`PondStore.human_action()` 在进程锁内读 JSON、调用引擎，并仅在 `ok=True` 时原子替换存档。平台模式下同样只在成功时更新 SQLite。两种宿主共享相同引擎判定。

## 8. 与平台 Adapter 的边界

### Adapter 负责

- 把 `eco_new`、`eco_observe`、`eco_act`、`eco_info`、`eco_save` 的结构化参数翻译为引擎指令；旧 `eco_play` 只保留兼容实现。
- 校验平台 `player_id`，以 `eco_sessions.player_id` 隔离玩家，保存完整 JSON 到 `data/sessions.db`。
- 维护最多 500 个会话和 30 天 TTL；用 `BEGIN IMMEDIATE` 覆盖“读取 → 运行 → 写回”整个事务。
- 因引擎使用进程级 `_STATE`，用 `_ENGINE_LOCK` 串行化引擎调用，并将 `engine.save_state` 替换为空函数，阻止平台请求写 `eco_save.json`。
- 提供平台 HTTP 所需的只读投影包装和 `human_action()`；处理平台公告/投票，这些不进入引擎规则。

### 引擎负责

- 所有数值、触发条件、随机调用顺序、指令效果、协作校验与状态迁移。
- `api_state/codex/folio/annals/species` 的字段语义。
- 生成 `pending_human_notices` 和年鉴记录，但不决定如何持久化或路由给哪个玩家。

### 边界上的维护规则

1. 新游戏规则写入 `engine.py`，不要在 adapter 或前端复制阈值。
2. 新状态字段同时加入 `fresh_state()`、`_migrate()`，必要时更新 full/lite 导入导出和 `api_*()`。
3. 新引擎命令若需平台调用，才在 adapter 增加 schema 与命令映射；只读展示优先扩展 `api_*()`。
4. 新人类协作动作同时更新 `_HUMAN_ACTIONS`、`api_state()` 可见状态、独立服务可用动作、平台路由和回归脚本。
5. 修改随机调用位置会改变所有后续确定性结果；必须运行天气随机流与协作 PRNG 校验。
6. 修改 `engine.py` 后运行 `python3 build_blind.py`，并用 `python3 build_blind.py --check` 确认生成物同步。
